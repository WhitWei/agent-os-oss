## 1. 平台是什么，不是什么

### 1.1 一句话定位

Agent OS 是一个**面向企业场景的 Agent 治理运行时**：它不负责"让 LLM 变聪明"，而是负责**约束和证明** LLM 驱动的 Agent 在你的企业系统里做的每一次高风险操作（写数据库、分配资产、审批流程）都经过了校验、留痕、可审计、可回滚决策。

如果把一个企业级 Agent 应用拆成两层：

```
┌─────────────────────────────────────┐
│  "让 Agent 聪明"的那一层              │  ← 你自己接入的 LLM（Claude / GPT 等）+ 你的业务 Prompt
│  (Agent OS 不提供，需要你自己接入)     │
├─────────────────────────────────────┤
│  "让 Agent 可信"的那一层              │  ← Agent OS 提供
│  (治理闸门 / 安全护栏 / 审批闭环 /     │
│   可观测性 / 测试纪律)                │
└─────────────────────────────────────┘
```

Agent OS 只负责下面这一层。这个定位选择是刻意的：市面上"让 Agent 更聪明"的框架（LangGraph、各种 Agent 编排框架）已经很多，但"企业客户签合同前会问的那些问题"——「数据校验规则谁定的」「敏感操作有没有人审」「出了事怎么查」「安全红线怎么证明测过」——大多数框架都没有现成答案。Agent OS 就是为了回答这些问题而存在的。

### 1.2 不是什么

- **不是一个开箱即用的 Agent 编排框架**：目前代码库里没有接入任何真实 LLM（Claude/GPT），聊天入口（`AgentOSKernel`）目前只是一个关键词路由的 MVP 占位实现（见 [11.3](#113-没有真实-llm-接入)）。你需要自己把 LLM 调用接进来。
- **不是一个"一键部署的 SaaS"**：这是一套你要 `pip install` 到自己项目里、按需组装的组件库 + 运行时骨架，不是托管服务。
- **不是企业级身份治理（IAM）平台**：Keycloak/OPA 这类企业 SSO/权限中枢集成属于路线图的 Track B，当前版本没有（见 [11.1](#111-没有企业身份治理-ssoopa)）。
- **不是一个通用工作流引擎**：内置的 SOP 引擎是为"LLM 生成数据 → 校验 → 人工审批 → 治理写入"这一类场景设计的，不是 BPMN 那种通用流程编排。

### 1.3 什么时候该用 Agent OS

✅ 适合：
- 你在给企业客户做一个会**真实修改企业数据**（IT 资产、HR 记录、财务台账、CRM 客户数据……）的 Agent 应用，客户会问"数据校验/审批/审计怎么保证"。
- 你是独立开发者/小团队，没有资源自建一整套 SHACL 校验 + nonce 防重放 + HITL 审批 + 可观测性 + 三层测试基础设施，想直接复用一套已经过红队级测试验证的骨架。
- 你需要给客户展示"这不是随便调个 API 的 demo，而是有治理闸门和测试证据的系统"。

❌ 不适合：
- 你只是想做一个不涉及企业系统写操作的纯问答/搜索类 Agent（不需要这套治理重量级设计）。
- 你需要企业级 SSO/RBAC 立即可用（当前版本没有，见路线图）。
- 你需要成熟的多 Agent 协作/A2A 协议互通（当前版本没有）。

## 3. 架构总览

### 3.1 分层视图

```
┌──────────────────────────────────────────────────────────────────┐
│ L0  渠道接入层  ChannelAdapter (Feishu / CLI / 自行扩展)            │
├──────────────────────────────────────────────────────────────────┤
│ L1  身份与权限层  AutonomyPolicy（最小权限：命令白名单/会话时长/     │
│                    写配额，声明式 YAML 配置）                        │
├──────────────────────────────────────────────────────────────────┤
│ L1.5 运行时安全护城河（嵌在 AgentOSKernel 的 pre/post-dispatch 钩子里）│
│      · SemanticFirewall（语义防火墙，正则级 prompt injection 拦截）  │
│      · CircuitBreaker（重复失败熔断，余弦相似度去重）                │
│      · BillingFuse（计费硬熔断，预算超支强制拒绝后续调用）           │
├──────────────────────────────────────────────────────────────────┤
│ L2  本体与知识治理层  Neo4j + Neosemantics + SHACL 强制断言校验       │
├──────────────────────────────────────────────────────────────────┤
│ L3  工具与治理写入层  MCP Governance Gateway（三段式写入工具）        │
├──────────────────────────────────────────────────────────────────┤
│ L4  编排与运行时层  SOPEngine（声明式 SOP YAML + HITL 审批状态机）    │
├──────────────────────────────────────────────────────────────────┤
│ L5  可观测与治理层  OpenTelemetry + 自托管 Langfuse + agent_feedback │
└──────────────────────────────────────────────────────────────────┘
```

### 3.2 ⚠️ 必须理解的架构现实：目前是"三个独立的组合根"，不是一个统一服务

这是本手册里最重要的一条架构说明，直接决定你该怎么把这套东西部署成一个真实能跑的服务。

当前代码库里，下面三个东西**分别可以独立运行，但彼此之间没有被任何一个 `main.py`/FastAPI 应用接起来**：

1. **`AgentOSKernel`**（聊天/webhook 入口，`src/zeroclaw/kernel.py`）——挂了防火墙/熔断器/计费熔断这三个安全钩子，但目前的意图识别只是关键词匹配（见 [11.3](#113-没有真实-llm-接入)），且**从未调用过 `SOPEngine`**。
2. **`SOPEngine`**（真正执行"校验 → 审批 → 治理写入"业务闭环的引擎，`src/workflow/sop_engine.py`）——这是承载真实业务价值的地方，但它只能通过 Python 代码直接实例化调用（参考 `scripts/run_uat.py`），**没有任何 HTTP 入口能触发它**，也没有任何路由把飞书审批卡片点击回调接到 `SOPEngine.resume()` 上。
3. **`GovernanceGateway`**（MCP 治理网关，`src/governance/mcp_server.py`）——把三段式写入暴露成 MCP 工具，供外部 LLM Agent（比如你在 Claude/GPT 里配置的 MCP 工具）调用，独立运行在自己的端口上。

**这意味着**：全仓库搜索找不到一行代码把"飞书用户发消息 → kernel 处理 → 触发 SOPEngine → 发审批卡片 → 用户点击 → 卡片回调 → SOPEngine.resume() → 治理写入"这条完整链路串成一个可对外提供服务的常驻进程。`scripts/run_uat.py` 和本手册第 6 章的教程走的都是"直接在 Python 里组装并调用"这条路径——这是当前唯一存在、且经过真实业务场景测试验证（见第 8 章）的集成方式。

**对你意味着什么**：如果你要把这套东西接入真实的飞书群/企业 IM，你需要自己写一个 FastAPI 应用把这三者粘合起来。第 9.3 节给出了一份可直接参考的最小实现骨架。这不是本手册的疏漏——这正是"超级个体/小团队"这类用户最需要提前知道的事实，好过部署到一半才发现缺了这一环。

### 3.3 技术栈

| 组件 | 选型 | 说明 |
|---|---|---|
| 运行时语言 | Python 3.11+ | 当前代码库是纯 Python 实现，没有任何 Rust/Go 组件 |

## 5. 核心概念详解

### 5.1 ZeroClaw Kernel：聊天入口 + 安全钩子

```python
kernel = AgentOSKernel(
    config=app_config,
    write_gate=write_gate,          # 可选：不传则无法处理 schema/write 相关意图
    autonomy_policy=autonomy_policy,  # 可选：不传则不做命令白名单/会话检查
)
response = await kernel.wake_up(message)  # message: ChannelMessage
```

`wake_up()` 内部按顺序执行：

1. **Pre-dispatch 钩子 1**：`SemanticFirewall.scan()` —— 正则级扫描用户输入，命中 prompt injection / 凭据泄露特征则拦截（`status: "intercepted"`）。
2. **Pre-dispatch 钩子 2**：`CircuitBreaker.is_open()` —— 如果同样的输入近期连续失败超过阈值，直接拒绝（`status: "tripped"`），不再重复尝试。
3. **Pre-dispatch 钩子 3**：`AutonomyPolicy` 检查（会话是否过期、疑似命令是否在白名单内）。
4. **业务处理**（当前是关键词路由 stub，见 [11.3](#113-没有真实-llm-接入)）。
5. **Post-dispatch 钩子**：命中 `validate/verify/write/execute` 等治理相关关键词时，记一笔计费（`BillingFuse.record_usage()`），累计超预算则后续调用一律拒绝（`status: "exhausted"`）。

失败路径会把失败特征反馈进熔断器（`CircuitBreaker.record_failure()`），供下次请求判断是否该拉闸。

### 5.2 AutonomyPolicy：声明式最小权限

配置文件：`src/policies/policy_config.yaml`

```yaml
filesystem:
  allowed_paths: ["/Users/you/agent-os-poc/**"]
  denied_paths: ["/etc/**", "**/.env"]
commands:
  mode: allowlist
  allowed: ["ls", "cat", "pytest", "python3", "git", "docker"]
write_quotas:
  max_operations_per_session: 50
  max_operations_per_minute: 10
  shacl_required_domains: ["it-asset-mgmt"]
session:
  max_duration_seconds: 3600
  max_token_budget_usd: 0.50
```

> ⚠️ **诚实说明**：`filesystem` 策略（`check_path_access`）目前**只是声明，没有被任何真实文件 IO 场景调用**——当前平台没有暴露"读写本地文件"这类工具，所以这条检查暂时是为未来接入文件类工具预留的。`commands`（会话检查）、`write_quotas`（写配额）已经在 `AgentOSKernel`/`WriteGate` 里真实生效。仓库自带的孤儿能力检测脚本（`scripts/check_wiring.py`）会持续跟踪这类"声明了但没接线"的能力，并要求显式登记理由，不允许静默存在。

### 5.3 MCP 治理网关：三段式写入闸门

这是整个平台安全模型的核心。任何一次"让 LLM 决定要写什么"到"数据真的写进数据库"之间，强制经过三步：

```
Stage 1  get_{domain}_schema
         → 返回该领域的 OWL 类定义 + SHACL 约束，告诉 LLM "应该长什么样"

Stage 2  verify_shacl_compliance_{domain}(data_rdf)
         → 用 pyshacl 对 LLM 组装出的 RDF 数据做强制断言校验
         → 校验通过才会返回一个签名的一次性 validation_nonce
         → nonce = HMAC-SHA256(timestamp + data_hash + domain)，绑定"这份数据"和"这次校验"

Stage 3  execute_governed_write_{domain}(data_rdf, validation_nonce)
         → 必须带着 Stage 2 拿到的 nonce 才能调用
         → nonce 会校验：签名合法性 / 是否过期(默认 300s) / 数据哈希是否与校验时一致(防篡改) / 是否已被消费过(防重放)
         → 通过后立即"燃烧"该 nonce（单次消费），再执行真实的 Cypher 写入
```

**已被红队级测试验证过的红线**（`tests/test_write_gate.py` 里的 `test_RED_LINE_*` 系列 + `tests/e2e` 里的 B1/B2 场景）：
- 不带 nonce 直接写 → 拒绝
- 伪造/挪用他人 nonce → 拒绝（数据哈希不匹配）
- 过期 nonce → 拒绝
- 数据被篡改后用原 nonce 写 → 拒绝
- 同一个 nonce 重放（哪怕数据完全一致）→ 拒绝（单次消费）

这三个工具默认通过 MCP `streamable-http` 协议暴露（端口见 `config.yaml` 的 `mcp.port`，默认 `8100`），可以直接配置进任何支持 MCP 的 LLM 客户端（Claude Desktop、Claude Code 等）作为外部工具使用。

### 5.4 本体与 SHACL：业务规则即代码

- **OWL 文件**（`docker/ontology/*.owl`）定义领域模型：有哪些类（Employee、HardwareAsset……）、类之间的关系（`assignedTo`、`manages`……）、属性的数据类型。
- **SHACL 文件**（`docker/ontology/*.shacl.ttl`）定义**强制断言**：哪些字段必填、类型是什么、枚举值范围是什么。SHACL 校验的是"这份数据是否合法"，不是"如何推理出新知识"——如果你的业务需要本体推理（比如"通过多层供应关系推断风险"），SHACL 本身不够用，需要额外接入推理引擎，这属于路线图范畴。
- **Ontology-as-Code CI**（`.github/workflows/ontology-ci.yml` + `scripts/check_ontology.py`）：任何 PR 修改了 OWL/SHACL 文件，自动跑 `tests/golden_dataset.py` 里定义的黄金数据集回归，防止"改本体的时候不小心放宽了某条校验规则"这种最容易被忽视、也最危险的安全退化。

### 5.5 SOPEngine：声明式业务流程 + 人机协同审批

一个 SOP 由 YAML 声明，支持四种步骤类型：

| 类型 | 作用 |
|---|---|
| `validate` | 对 `ctx.data` 里某个字段做 SHACL 校验，通过后把 nonce 存回上下文 |
| `human_approval` | 按 `condition` 表达式判断是否需要人工审批；需要则发送交互卡片并**挂起**整个流程，等待外部回调 `resume()` |
| `action` (`governed_write`) | 用之前 `validate` 步骤拿到的 nonce，执行真正的治理写入 |
| `notify` | 发送通知，不等待回复 |

```yaml
- id: approve_sensitive_asset
  type: human_approval
  condition: "sensitivity_level in ['HIGH', 'CRITICAL']"   # ⚠️ 必须用列表字面量，见下方提示框
  card_title: "🔐 敏感设备分配 — 需要您的审批"
  card_message: "..."
  timeout_seconds: 3600
```

> ⚠️ **条件表达式语法提示**：`condition` 用 [`simpleeval`](https://github.com/danthedeckie/simpleeval) 的 `EvalWithCompoundTypes` 求值，**必须用列表 `[...]` 或元组 `(...)` 字面量**（两者均受支持），变量只能来自 `ctx.data`。求值失败会默认按"需要审批"处理（失败即从严），不会静默放行。

流程状态机（`SOPRunState`）：`PENDING → RUNNING → (SUSPENDED → APPROVED/REJECTED) → COMPLETED/FAILED`。挂起状态会持久化到 SQLite（`WorkflowStateStore`），**服务重启后可以用同一个 `run_id` 从磁盘正确恢复**——这一点已经过真实场景测试（`tests/e2e/test_c_resilience_scenarios.py`）。

每一次审批决策（批准/驳回 + 理由 + 审批人）都会独立落入 `agent_feedback` 表，与 `run_id` 绑定，是未来做规则迭代、行为调优的核心数据源。

### 5.6 运行时安全护城河

| 组件 | 触发方式 | 默认行为 |
|---|---|---|
| `SemanticFirewall` | 正则匹配已知 prompt injection / 角色扮演越权 / 凭据泄露特征 | 命中 `high`/`critical` 严重度即抛 `SecurityInterceptError` |
| `CircuitBreaker` | 同一输入（按参数哈希 + 结构指纹相似度）连续失败达阈值（默认 3 次/300s 窗口） | 拉闸，后续同类调用直接拒绝，窗口过期自动重置 |
| `BillingFuse` | 每次触发计费的调用累计 token 成本 | 超过预算上限（默认 $0.50/会话）后续调用全部拒绝，可注册 `revoke_callback` 联动吊销凭据 |
| `WasmSandbox` | 需要执行不受信代码时手动调用 | 16MB 内存硬限、CPU 指令数硬限、无文件/网络访问、敏感路径预扫描 |

> ⚠️ `WasmSandbox` 目前**没有被任何工具调用点接入**——它是为未来"执行第三方不受信插件代码"这类功能预置的基础设施，当前平台没有暴露这样的工具。同样由 `scripts/check_wiring.py` 持续跟踪并要求显式登记。

### 5.7 可观测性

- 主链路走 OpenTelemetry → OTLP Collector → 自托管 Langfuse（`docker/docker-compose.yml` 已包含 Langfuse Web/Worker + PostgreSQL + Redis）。
- 安全拦截事件（`security.intercept` span）、SHACL 校验失败事件（`shacl.validation_error` span）作为独立的自定义 span 类型出现在 Trace 里（`src/observability/security_dimensions.py`），可以在 Langfuse 里单独筛选/告警，而不是淹没在普通日志中。

## 6. 手把手教程：接入你自己的业务场景

以下以"把 IT 资产入职场景改造成你自己的业务领域"为例，假设你要做一个**销售合同审批**场景。

### 第 1 步：定义 OWL 本体

新建 `docker/ontology/sales-contract.owl`，定义你的类和属性（参考 `docker/ontology/it-asset-mgmt.owl` 的写法）：

```turtle
@prefix owl: <http://www.w3.org/2002/07/owl#> .
@prefix rdfs: <http://www.w3.org/2000/01/rdf-schema#> .
@prefix xsd: <http://www.w3.org/2001/XMLSchema#> .
@prefix sc: <http://your-company.local/ontology/sales-contract#> .

sc:Contract rdf:type owl:Class ;
    rdfs:label "Sales Contract"@en .

sc:contractAmount rdf:type owl:DatatypeProperty ;
    rdfs:domain sc:Contract ; rdfs:range xsd:decimal .

sc:customerName rdf:type owl:DatatypeProperty ;
    rdfs:domain sc:Contract ; rdfs:range xsd:string .
```

### 第 2 步：定义 SHACL 强制断言

新建 `docker/ontology/sales-contract.shacl.ttl`：

```turtle
sc:ContractShape
    rdf:type sh:NodeShape ;
    sh:targetClass sc:Contract ;
    sh:property [
        sh:path sc:contractAmount ;
        sh:datatype xsd:decimal ;
        sh:minCount 1 ; sh:maxCount 1 ;
        sh:name "contractAmount" ;
    ] ;
    sh:property [
        sh:path sc:customerName ;
        sh:datatype xsd:string ;
        sh:minCount 1 ; sh:maxCount 1 ;
        sh:name "customerName" ;
    ] .
```

### 第 3 步：注册到 `config.yaml`

```yaml
ontology:
  domains:
    - name: "sales-contract"
      owl_file: "sales-contract.owl"
      shacl_file: "sales-contract.shacl.ttl"
```

### 第 4 步：给关键规则写黄金数据集回归用例

在 `tests/golden_dataset.py` 里补充你的 `GoldenCase`（合规样例 + 违规样例各至少一条），这样以后任何人改动 SHACL 文件，CI 会自动验证没有削弱你的核心业务约束。

### 第 5 步：写 SOP YAML

新建 `src/workflow/sop_examples/contract-approval.sop.yaml`（结构完全参考 `it-onboarding.sop.yaml`）：

```yaml
sop_id: "contract-approval-v1"
name: "销售合同审批"
steps:
  - id: validate_contract
    type: validate
    domain: "sales-contract"
    data_ref: "contract_ttl"

  - id: approve_large_contract
    type: human_approval
    condition: "contract_amount > 100000"
    card_title: "💰 大额合同审批"
    card_message: "客户 {customer_name}，金额 {contract_amount}，请审批。"

  - id: execute_write
    type: action
    action_type: "governed_write"
    domain: "sales-contract"
    data_ref: "contract_ttl"

  - id: notify_done
    type: notify
    notify_message: "✅ 合同 {customer_name} 已完成审批并归档。"
```

### 第 6 步：组装并跑通（参考 `scripts/run_uat.py` 的组合根模式）

```python
from agentos_kernel.config import ConfigLoader
from governance.schema_provider import SchemaProvider
from governance.write_gate import WriteGate
from governance.neo4j_client import Neo4jClient
from database.feedback_db import FeedbackDB
from database.state_store import WorkflowStateStore
from workflow.sop_engine import SOPEngine

config = ConfigLoader("config.yaml").load()
sp = SchemaProvider(config.ontology.owl_dir, config.ontology.shacl_dir, config.ontology.domains)
neo4j_client = Neo4jClient(config.neo4j)
gate = WriteGate(sp, neo4j_client, config.mcp.validation.nonce_secret, config.mcp.validation.nonce_ttl_seconds)
engine = SOPEngine(
    schema_provider=sp, write_gate=gate,
    feedback_db=FeedbackDB("prod_feedback.db"),
    state_store=WorkflowStateStore("prod_state.db"),
)

sop = SOPEngine.load_sop("src/workflow/sop_examples/contract-approval.sop.yaml")

# 你自己的 LLM/业务逻辑组装出 contract_ttl，并预先跑一次 SHACL 校验拿到 nonce
report, nonce = gate.verify_shacl_compliance("sales-contract", contract_ttl)
assert report.is_valid

ctx = engine.create_run(sop, {
    "contract_ttl": contract_ttl,
    "customer_name": "某某科技",
    "contract_amount": 150000,
    "_nonce_contract_ttl": nonce,
})
ctx = await engine.run(sop, ctx)
# ctx.state 可能是 SUSPENDED（等待审批）或 COMPLETED（无需审批直接完成）
```

### 第 7 步：把审批回调接回来

飞书交互卡片被点击后，你的 webhook 收到回调，用 `FeishuAdapter.parse_card_callback(body)` 解析出 `run_id`/`decision`/`reviewer`/`reason`，再调用：

```python
ctx = state_store.load_state(run_id)
ctx = await engine.resume(sop, ctx, decision=decision, reason=reason, approver_id=reviewer)
```

> 提醒：这一步目前**没有现成的 HTTP 路由帮你接好**——见 [3.2 节](#32-必须理解的架构现实目前是三个独立的组合根不是一个统一服务)和 [9.3 节](#93-补齐生产入口参考实现) 的参考实现。

### 第 8 步：为你的场景补一套三层测试

参考 `tests/e2e/test_a_onboarding_business_flow.py` 的写法，至少覆盖：正向完成路径、审批驳回路径、条件自动跳过路径、数据不合规路径。这是本平台对自己的要求，也强烈建议你对自己的场景一视同仁（见第 8 章）。

## 8. 三层测试防御体系（平台自带的工程纪律）

这套测试哲学是本平台在真实迭代中踩过坑之后沉淀出来的，也是向企业客户证明"这套系统是被真正测试过的"的核心论据，建议你在自己的业务场景开发中原样照搬。

### 8.1 为什么需要三层，而不是"多写点单测"

单测（组件隔离测试，外部依赖全 Mock）和端到端冒烟测试（只测最外层 happy path）中间，存在一个系统性盲区：**"组件 A 的单测全绿，组件 B 的单测也全绿，但主流程里 A 其实从没真正调用过 B"**。本仓库在真实迭代中至少两次踩中同一类缺陷：

1. `WriteGate.execute_governed_write` 从 `async def` 改造时，`SOPEngine` 里唯一的调用方没有跟着改成 `await`，导致治理写入变成一个从未被执行的协程——`ctx.state` 照样汇报 `COMPLETED`，Neo4j 里其实什么都没写进去。
2. `SemanticFirewall`/`CircuitBreaker`/`BillingFuse` 三个安全模块各自单测齐全，但很长一段时间里**没有一行代码把它们接进 `AgentOSKernel.wake_up()`**——单独调用能拦截，走真实入口完全不设防。

这两类缺陷都是"单测测不出来，端到端冒烟测试也测不出来（因为冒烟测试通常只覆盖一条 happy path）"的——只有专门针对"调用链路是否真的接通"和"多种业务分支是否都被真实数据验证过"设计的测试才能抓到。

### 8.2 三层定义

| 层级 | 目录 | 验证的问题 | Mock 边界 |
|---|---|---|---|
| **L1 单元测试** | `tests/*.py` | 单个模块自身逻辑对不对 | 外部物理依赖（Neo4j、网络）全部 Mock/None |
| **L2 集成/接线测试** | `tests/integration/` | 组件是否真的被主入口调用、真实依赖是否真的生效 | 用 `testcontainers` 拉起真实 Neo4j；**不 mock 内部安全/治理组件**；只在最外层打桩 |
| **L3 真实业务场景测试** | `tests/e2e/` | 一个完整业务叙事，在真实依赖下是否真的按预期发生（正向路径 + 拒绝路径 + 安全红线 + 灾难恢复 + 并发隔离） | 100% 真实组合根调用，用独立的数据库读回校验结果，不信任被测代码自己返回的状态字段 |

配套的自动化门禁：

- **`scripts/check_wiring.py`**（孤儿能力检测）：静态扫描声明的安全/治理能力是否在生产入口代码里被真实调用，未登记的"孤儿能力"直接 CI 失败，倒逼每个新组件从第一天起就想清楚"谁来调用它"。
- **`.github/workflows/integration-ci.yml`**：`wiring-audit` / `l2-integration-tests` / `l3-business-scenarios` / `unit-tests` 四个独立 job，任何一层失守都会挡住合并。
- **`.github/workflows/ontology-ci.yml`**：本体/SHACL 文件的黄金数据集回归。

### 8.3 运行方式

```bash
pytest tests/ -v                                   # 全量（需要 Docker）
pytest tests/ -v --ignore=tests/integration --ignore=tests/e2e   # 仅 L1，无需 Docker
pytest tests/integration -v                         # 仅 L2
pytest tests/e2e -v                                 # 仅 L3
python3 scripts/check_wiring.py                     # 孤儿能力静态检测
```

### 9.1 `docker-compose.yml` 服务清单

### 9.2 生产环境必须修改的默认值清单

| 项 | 默认值 | 生产要求 |
|---|---|---|
| `MCP_NONCE_SECRET` | `dev-nonce-secret-change-in-prod` | 必须换成高熵随机值，且不同环境（dev/staging/prod）不复用 |
| Neo4j 密码 | `agentos123` | 换成随机密码，走 secret manager 注入 |

| `write_quotas.max_operations_per_session/minute` | 50 / 10 | 按业务并发量评估 |

### 9.3 补齐生产入口（参考实现）

如第 3.2 节所述，当前仓库**没有**一个把 kernel + SOPEngine + 飞书回调粘合起来的 HTTP 服务入口。下面是一份可以作为起点的参考骨架（不是仓库自带代码，需要你根据自己的鉴权/路由需求补全）：

```python
# main.py（参考骨架，需要按需补全）
from fastapi import FastAPI, Request
from agentos_kernel.config import ConfigLoader
from agentos_kernel.kernel import AgentOSKernel
from adapters.feishu_adapter import FeishuAdapter
from workflow.sop_engine import SOPEngine
# ... 按第 6 章的方式组装 schema_provider / write_gate / feedback_db / state_store / sop_engine

app = FastAPI()
config = ConfigLoader("config.yaml").load()
feishu = FeishuAdapter(config.adapters.feishu, message_handler=kernel.wake_up)

@app.post(config.adapters.feishu.webhook_path)
async def feishu_webhook(request: Request):
    body = await request.json()

    # 1) URL 校验挑战
    challenge = feishu.verify_challenge(body)
    if challenge:
        return challenge

    # 2) 交互卡片审批回调 → 路由给 SOPEngine.resume()
    if "action" in body:
        callback = feishu.parse_card_callback(body)
        if callback:
            ctx = state_store.load_state(callback["run_id"])
            await sop_engine.resume(
                sop, ctx, decision=callback["decision"],
                reason=callback["reason"], approver_id=callback["reviewer"],
            )
            return {"status": "ok"}

    # 3) 普通消息 → 触发你自己的业务逻辑（LLM 决策是否发起 SOP，或走 kernel.wake_up 走简单意图）
    message = feishu.parse_event(body)
    if message:
        response = await kernel.wake_up(message)
        await feishu.send_response(response)
    return {"status": "ok"}
```

补全这份骨架前，务必：签名校验（`feishu.verify_signature`）要在生产环境强制开启；`run_id → SOPDefinition` 的映射需要你自己维护（当前 `state_store` 只存 `SOPRunContext`，不存 `SOPDefinition` 本身）；建议按 [8.2 节](#82-三层定义) 的模式为这份骨架补一套 L3 测试再上生产。

### 10.2 尚未覆盖，生产前需要你自行评估/补齐

## 11. 已知限制与路线图

### 11.1 没有企业身份治理（SSO/OPA）

`AutonomyPolicy` 是进程内声明式策略，不对接企业 SSO/AD，也没有 OPA 这类细粒度策略引擎。路线图上作为独立的"追加层"引入，不影响已落地的核心链路——见 `Agent_OS方案融合分析与修订实施计划.md` 里 Track B 的规划。

### 11.2 没有跨 Agent 协作协议

当前不涉及 A2A（Google/Linux Foundation 标准）或其他跨 Agent 通信协议。如果你需要对接 Salesforce/ServiceNow 等外部 Agent 生态，需要自行开发协议网桥。

### 11.3 没有真实 LLM 接入

`AgentOSKernel._process_message` 目前是纯关键词匹配的 MVP 占位实现，不调用任何 LLM。这意味着：
- "Agent 能自己判断该走哪个 SOP、该怎么组装 RDF 数据"这件事目前需要你自己在应用层实现（调用 Claude/GPT，用它生成符合 `get-schema` 返回结构的数据，再走 `verify_shacl_compliance`）。
- MCP 治理网关（第 5.3 节）就是设计给你接入的 LLM 调用的标准入口——你的 LLM 侧只需要知道这三个工具，不需要知道底层校验细节。

### 11.4 三个组合根尚未合并成一个服务

见 [3.2 节](#32-必须理解的架构现实目前是三个独立的组合根不是一个统一服务)。

### 11.5 本体推理能力边界

SHACL 只做约束校验（"这份数据合不合法"），不做 OWL 推理（"从已知事实推导新结论"）。如果业务需要复杂的传递性/继承性推理，需要额外引入推理引擎，SHACL 本身不提供。

## 13. 常见问题 FAQ

**Q: 为什么不直接用 LangGraph/其他 Agent 框架？**
A: 它们解决的是"编排复杂度"问题（多轮状态机、多智能体协作），Agent OS 解决的是"治理可信度"问题（数据校验、审批留痕、安全护栏、测试证据）。两者不冲突——如果你的场景确实需要复杂的多步骤 LLM 推理编排，完全可以在 Agent OS 的治理闸门之上叠加 LangGraph 之类的编排层，把 `verify_shacl_compliance`/`execute_governed_write` 当作 LangGraph 图里的受治理工具节点来用。

**Q: 我必须用 Neo4j 吗？能不能换成 PostgreSQL/MySQL？**
A: SHACL 校验本身只依赖 `rdflib` + `pyshacl`，不强绑定 Neo4j——`WriteGate.verify_shacl_compliance` 阶段完全不接触数据库。真正绑定 Neo4j 的只有 Stage 3 的 `_rdf_to_cypher`/`execute_governed_write` 落库逻辑。如果你要换成关系型数据库，需要自己重写这部分 RDF→SQL 的转换逻辑，SHACL 校验和 nonce 治理机制可以原样复用。

**Q: 测试要求有 Docker，我在 CI 之外的环境没有 Docker 怎么办？**
A: `tests/integration` 和 `tests/e2e` 里的测试会检测 Docker 是否可用，不可用时自动 `pytest.skip`（不是失败），`tests/` 根目录的 150 条 L1 单元测试不受影响，可以独立跑：`pytest tests/ --ignore=tests/integration --ignore=tests/e2e`。

**Q: SOP 的 `condition` 表达式为什么要用列表而不是元组？**
A: 两者当前实现都支持（`EvalWithCompoundTypes` 均支持 list/tuple 字面量）。历史上早期版本用的是 `simpleeval.simple_eval()` 便捷函数，它出于安全考虑禁用了所有复合类型字面量，用元组或列表都会报错并被静默吞掉，导致条件永远判定为"需要审批"。这个问题已经修复，但如果你看到类似 `Sorry, ... is not available in this evaluator` 的告警日志，说明某处代码又退回到了 `simple_eval()`，需要检查是否误用。
